Request Flow Summary

FlowPath
IM MessageIM Platform → Platform Adapter → Message Handler → Router.GetOrCreate → Process.Send → EventLog → Reply to IM
Dashboard ChatBrowser → WS Hub / REST API → Router.GetOrCreate → Process.Send → EventLog → WS Push to Browser
DiscoveryBrowser → GET /api/discovered → Scan ~/.claude → Render in sidebar
TakeoverBrowser → POST /api/discovered/takeover → Kill external → Router.Takeover(--resume) → LoadHistory → Managed Session
Cron JobTimer fires → Scheduler → Router.GetOrCreate → Process.Send → Reply to IM

Process Lifecycle

StateTriggerDescription
SpawningGetOrCreate / TakeoverProcess starting, protocol Init handshake
ReadyInit complete / Send completeWaiting for next message
RunningSend() calledProcessing user message, events flowing
DeadExit / Timeout / KillNext message triggers resume via --resume {sessionID}
SuspendedService restartLoaded from session store, no process. Resume on next message

Core Design Principles

PrincipleImplementation
Zero External DependenciesAll state file-based (JSON). No Redis, no DB, no MQ.
Protocol AbstractionProtocol interface: Claude (stream-json) or Kiro (JSON-RPC 2.0)
Long-lived ProcessesCLI subprocess stays open across turns. stdin/stdout stay connected.
Session ResumabilityDead sessions resume via --resume {sessionID}. History loaded from JSONL.
Unified ManagementDashboard sees all sessions (IM, dashboard, takeover) with real-time events.
Multi-NodeNodeClient + WS Relay aggregates remote naozhi instances into one dashboard.
